STEP 1

.env ファイル保護（permissions.deny 追加）

必須
△ 部分適用 — 追加が必要

現在の permissions.deny には Read(~/.env) のみ設定されていますが、プロジェクト内の .env ファイルや *.key / *.pem は保護されていません。

Claude Code の内蔵ツール（Read・Glob・Edit・Write・Bash の cat / grep 等）による機密ファイルへのアクセスを拒否するルールを追加します。

⚠️

sandbox との違い： sandbox は Bash ツールの実行のみに適用されます。Read / Edit / Glob / Write ツール経由のアクセスは permissions.deny でのみ制限できます。両方設定することで多層防御になります。

追加する deny ルール

~/.claude/settings.json の permissions.deny 配列に以下を追加します。

~/.claude/settings.json — permissions.deny に追加JSON
// 以下を permissions.deny 配列に追記

// --- .env / 秘密鍵ファイル保護 ---
"Read(.env*)",
"Read(secrets*)",
"Read(*.key)",
"Read(*.pem)",
"Glob(.env*)",
"Glob(secrets*)",
"Glob(*.key)",
"Glob(*.pem)",
"Edit(.env*)",
"Edit(secrets*)",
"Edit(*.key)",
"Edit(*.pem)",
"Write(.env*)",
"Write(secrets*)",
"Write(*.key)",
"Write(*.pem)",
"Bash(cat .env*)",
"Bash(cat *secrets*)",
"Bash(grep * .env*)",
"Bash(less .env*)",
"Bash(more .env*)",
"Bash(head .env*)",
"Bash(tail .env*)"

以下のコマンドで現在の deny 設定を確認できます。

現在の deny ルール確認WSL Ubuntu
cat ~/.claude/settings.json | python3 -m json.tool | grep -A50 '"deny"'
STEP 2

file-history 自動削除（cron）

必須
✗ 未設定

Claude Code は編集バックアップを ~/.claude/file-history/ に自動蓄積します。放置するとディスク使用量が数十 MB 以上になる場合があります。毎日 0 時に自動削除する cron ジョブを設定します。

現在のサイズ確認
file-history のサイズ確認WSL Ubuntu
du -sh ~/.claude/file-history/ 2>/dev/null || echo "ディレクトリなし"
手動削除
手動で即座に削除WSL Ubuntu
find ~/.claude/file-history -mindepth 1 -delete
cron ジョブの設定
crontab を編集WSL Ubuntu
crontab -e

エディタが開いたら末尾に以下を追加します。

cron エントリ（毎日 0:00 に自動削除）crontab
0 0 * * * find $HOME/.claude/file-history -mindepth 1 -delete >> /tmp/file-history-cleanup.log 2>&1
設定確認WSL Ubuntu
crontab -l
ℹ️

WSL2 環境では cron デーモンが自動起動しない場合があります。sudo service cron start で起動するか、/etc/wsl.conf に自動起動設定を追記してください。

WSL2 で cron を自動起動（/etc/wsl.conf に追記）/etc/wsl.conf
[boot]
command = service cron start
STEP 3

dotenvx による .env ファイル暗号化

任意
✗ 未設定

dotenvx は .env ファイルをコミット前に暗号化するツールです。暗号化済みの .env ファイルはリポジトリに含められ、複号鍵は .env.keys に分離管理されます。

ℹ️

STEP 1 の permissions.deny 設定のみでも .env 保護は機能します。dotenvx は Git リポジトリで .env を共有・管理する場合に有効です。

インストール（Linux / WSL2）
dotenvx インストール（Linux x64）WSL Ubuntu
curl -fsSL https://github.com/dotenvx/dotenvx/releases/latest/download/dotenvx-linux-x64.tar.gz \
  -o /tmp/dotenvx.tar.gz \
  && tar xzf /tmp/dotenvx.tar.gz -C /tmp/ \
  && mv /tmp/dotenvx ~/.local/bin/
dotenvx --version
使い方
.env を暗号化（プロジェクトルートで実行）WSL Ubuntu
dotenvx encrypt
# .env が暗号化され、複号鍵が .env.keys に書き出される
# .env.keys は .gitignore に追加し、安全な場所で保管すること
暗号化済み .env を使ってコマンドを実行WSL Ubuntu
dotenvx run -- node server.js
.gitignore に .env.keys を追加.gitignore
.env.keys

STEP 5

CLAUDE.md セキュリティルール

必須
✗ 未設定

~/.claude/CLAUDE.md に記述したルールは全セッションで自動的に読み込まれ、Claude Code の動作方針として機能します。セキュリティポリシーを明文化しておくことで、意図しない操作を防ぎます。

ディレクトリ作成と CLAUDE.md の生成
~/.claude/CLAUDE.md を作成WSL Ubuntu
mkdir -p ~/.claude
cat > ~/.claude/CLAUDE.md << 'EOF'
# セキュリティルール

## 機密ファイルの取り扱い
- `.env*`, `secrets*`, `*.key`, `*.pem` ファイルは読み取り・編集・表示を行わない
- `~/.ssh/`, `~/.aws/`, `~/.gnupg/` 以下のファイルは操作しない
- APIキーやトークンをコード・コメント・ログに含めない

## 破壊的操作の禁止
- `rm -rf` は明示的な確認なしに実行しない
- `git push --force` / `git reset --hard` は確認後のみ実行する
- `sudo` コマンドは最小権限の原則に従い、不要な場合は使用しない

## ネットワーク・外部通信
- 許可ドメイン以外への通信が必要な場合は事前に確認する
- 認証情報を外部サービスに送信しない

## コード品質
- テストなしで本番コードを変更しない（既存テストがある場合）
- セキュリティに関わるコード変更は必ず変更内容を説明する
EOF
echo "CLAUDE.md を作成しました"
作成確認WSL Ubuntu
cat ~/.claude/CLAUDE.md
STEP 6

TDD モード（oh-my-claudecode プラグイン）

任意
✗ 未設定

oh-my-claudecode プラグインの TDD モードを有効にすると、コード変更時に Red → Green → Refactor サイクルを強制し、カバレッジ 80% 以上を維持するよう Claude Code が自律的に動作します。

⚠️

このプラグインは外部 Git リポジトリからキャッシュされます。プラグインのコードが sandbox 外で動作する点に注意してください。信頼できるソース以外のプラグインは使用しないでください。

settings.json への追加

以下を ~/.claude/settings.json に追加します。

~/.claude/settings.json — enabledPlugins に追加JSON
"enabledPlugins": {
  // 既存のプラグイン設定を保持しつつ追加
  "oh-my-claudecode@omc": true
}
TDD モードの使い方
  • 1.Claude Code を起動後、/omc tdd または TDD モードを明示的に指示する
  • 2.Red フェーズ：失敗するテストを先に書く
  • 3.Green フェーズ：テストをパスする最小実装を書く
  • 4.Refactor フェーズ：コードを整理する（テストはパスしたまま）
STEP 7

自動コンパクト設定（PreToolUse フック）

任意
✗ 未設定

コンテキスト使用量が閾値（デフォルト 50%、推奨 25%）を超えると自動的にチェックポイントを保存し、コンパクションの前に進捗を記録します。長時間のセッションで作業内容を失わないための安全策です。

チェックポイントスクリプトの作成
スクリプトディレクトリを作成WSL Ubuntu
mkdir -p ~/.claude/scripts
~/.claude/scripts/save-progress.sh を作成~/.claude/scripts/save-progress.sh
#!/bin/bash
# コンパクション前に進捗チェックポイントを保存するスクリプト

CHECKPOINT_FILE="$HOME/.claude/progress-checkpoint.md"
TIMESTAMP=$(date '+%Y-%m-%d %H:%M:%S')
SESSION_ID="${CLAUDE_SESSION_ID:-unknown}"
PROJECT_NAME=$(basename "$PWD")

cat >> "$CHECKPOINT_FILE" << EOF

## チェックポイント: $TIMESTAMP
- セッションID: $SESSION_ID
- プロジェクト: $PROJECT_NAME
- 作業ディレクトリ: $PWD
EOF

echo "チェックポイントを保存しました: $TIMESTAMP"
スクリプトに実行権限を付与WSL Ubuntu
chmod +x ~/.claude/scripts/save-progress.sh
settings.json への hooks 追加

以下を ~/.claude/settings.json に追加します。

~/.claude/settings.json — hooks セクションを追加JSON
"hooks": {
  "PreToolUse": [
    {
      "matcher": "Bash",
      "hooks": [
        {
          "type": "command",
          "command": "bash ~/.claude/scripts/save-progress.sh",
          "environment": {
            "COMPACT_THRESHOLD": "25"
          }
        }
      ]
    }
  ]
}
ℹ️

COMPACT_THRESHOLD=25 はコンテキスト使用量 25% でチェックポイントを保存します。デフォルト (50) より早めに保存するため、長時間セッションで安全に使えます。
