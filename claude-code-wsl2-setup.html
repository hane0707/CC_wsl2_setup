前提

前提確認・ネイティブインストール済みの場合

⚠️

ネイティブインストール済みの場合：WSL2側とネイティブWindows側はそれぞれ独立したバイナリです。競合は起きませんが、PATH混入に注意してください。本手順のSTEP 3・4・6に追加の注記があります。

セットアップを始める前に、各コンポーネントの関係を把握しておきましょう。

アーキテクチャ概要

以下を確認してから進めてください。

確認項目 要件 確認方法
Windows バージョン Windows 10 21H2 以上、または Windows 11 Win + R → winver
CPU 仮想化 Virtualization: Enabled タスクマネージャー → パフォーマンス → CPU
空きメモリ 8GB 以上推奨（最低 4GB） タスクマネージャー → パフォーマンス → メモリ
空きディスク 10GB 以上 エクスプローラー → PC
Claude アカウント Pro / Max / Teams / Enterprise / Console のいずれか 無料プランは不可
STEP 1

WSL2 インストール

必須
✅

この手順はすでに完了しています。 ログで yourname@hostname:~$ が表示されUbuntuに入れているので、WSL2のインストールは成功しています。STEP 2 に進んでください。

以下は記録として残します。

PowerShell（管理者）Windows
wsl --install -d Ubuntu

再起動後、自動的に Ubuntu のセットアップが始まります。ユーザー名・パスワードを設定して完了です。

⚠️

よくある誤操作： Ubuntu内（プロンプトが yourname@hostname:~$ の状態）で wsl shutdown と打っても「command not found」になります。wsl --shutdown は PowerShell から実行するコマンドです。Ubuntu から抜けるには exit を使います。

プロンプトの見た目で今どちらにいるかが分かります。

Ubuntu 内（WSL）のプロンプト — Linux コマンドのみ有効WSL Ubuntu
yourname@hostname:~$
PowerShell のプロンプト — ここで wsl --shutdown を実行するWindows
PS C:\Users\yourname>
ℹ️

STEP 2 の wsl --shutdown について： Ubuntu タブを exit で閉じてからPowerShellで実行してください。または Windows Terminal で PowerShell タブを新たに開いて実行するのが確実です。

STEP 2

.wslconfig でメモリ上限を設定

必須

Claude Code は放置するとメモリを大量消費することがあります。WSL2 全体の上限を事前に設定します。

エクスプローラーで C:\Users\<あなたのユーザー名>\ を開き、.wslconfig というファイルを新規作成して以下を記入します。

.wslconfig（新規作成）C:\Users\YourName\.wslconfig
[wsl2]
memory=8GB
swap=16GB

[experimental]
autoMemoryReclaim=gradual

memory は搭載 RAM の 50% が目安です（16GB 搭載なら 8GB）。

保存後、UbuntuタブをすべてexitまたはタブのXで閉じてから、PowerShell で以下を実行して設定を反映します。

PowerShell（Ubuntuを閉じた後に実行）Windows
wsl --shutdown
ℹ️

何も表示されなければ成功です。「致命的なエラー」が出た場合はすでに停止済みなので無視して次へ進んでください。

ℹ️

wsl --shutdown はWindowsのコマンドです。Ubuntu内（yourname@hostname:~$ の状態）では動きません。必ずPowerShellのプロンプト（PS C:\...）から実行してください。

STEP 3

Ubuntu 初期設定

必須

Windows Terminal で Ubuntu タブを開きます。タブバー右端の ∨ をクリックして「Ubuntu」を選ぶか、PowerShell で wsl と入力すると入れます。プロンプトが yourname@hostname:~$ のような表示になればOKです。

パッケージ更新WSL Ubuntu
sudo apt update && sudo apt upgrade -y
⚠️

ネイティブインストール済みの場合：WSL内でも PATH にWindowsの claude.exe が混入することがあります。以下で確認してください。

PATH 汚染の確認（/mnt/c/ が出れば汚染あり）WSL Ubuntu
echo $PATH | tr ':' '\n' | grep -i mnt

/mnt/c/ が出た場合は汚染ありです。以下を実行して ~/.bashrc にフィルタを追記します。

PATH フィルタを .bashrc に追記WSL Ubuntu
echo 'export PATH=$(echo $PATH | tr ":" "\n" | grep -v "/mnt/c" | tr "\n" ":" | sed "s/:$//")' >> ~/.bashrc
.bashrc を反映WSL Ubuntu
source ~/.bashrc
ℹ️

PATH汚染がない（/mnt/c/ が出ない）場合はこの操作は不要です。次のステップに進んでください。

STEP 4

Claude Code インストール（WSL内）

必須
ℹ️

ネイティブインストール済みの場合：WSL内への追加インストールは問題ありません。Windows側のバイナリ（~\.local\bin\claude.exe）とWSL側のバイナリは完全に独立しています。設定ファイル（~/.claude/）は認証情報を含め共有されません。WSL内で別途ログインが必要です（STEP 6）。

公式推奨のネイティブインストーラーを使います（Node.js 不要）。

PATH を反映WSL Ubuntu
source ~/.bashrc

バージョン番号が表示されれば成功です。

バージョン確認WSL Ubuntu
claude --version

npm install -g @anthropic-ai/claude-code は非推奨（公式が廃止予定）。上記の curl コマンドを使ってください。

STEP 5

サンドボックス有効化 & settings.json 設定

推奨

WSL2 限定の機能です。Claude Code が Bash ツールで実行するコマンドをサンドボックス内に隔離し、意図しないネットワーク通信やシステム変更を防ぎます。

ℹ️

サンドボックスの適用範囲： Bash ツールの実行のみに適用されます。Read / Write / WebSearch / WebFetch / MCP ツールやフック（Hooks）には適用されません。ファイル操作の制限は別途 permissions で設定します。

5-1 依存パッケージのインストール
bubblewrap + socat インストールWSL Ubuntu
sudo apt install -y bubblewrap socat
5-2 settings.json の作成（推奨設定）

公式リポジトリには settings-lax / settings-bash-sandbox / settings-strict の3プリセットがあります。個人のWSL2開発環境には settings-bash-sandbox をベースにした以下の設定が適しています。設定は ~/.claude/settings.json（全プロジェクト共通）に置きます。プロジェクト単位で上書きしたい場合は .claude/settings.json 、適用前にテストしたい場合は .claude/settings.local.json を使ってください。

設定ディレクトリの作成WSL Ubuntu
mkdir -p ~/.claude
settings.json の作成WSL Ubuntu
cat > ~/.claude/settings.json << 'EOF'
{
  "dangerouslySkipPermissions": "disable",
  "enableAllProjectMcpServers": false,
  "sandbox": {
    "enabled": true,
    "autoAllowBashIfSandboxed": false,
    "allowUnsandboxedCommands": false,
    "network": {
      "allowedDomains": [
        "api.anthropic.com",
        "github.com",
        "registry.npmjs.org",
        "pypi.org"
      ],
      "allowUnixSockets": [],
      "allowAllUnixSockets": false,
      "allowLocalBinding": true
    },
    "filesystem": {
      "denyRead": [
        "~/.ssh/**",
        "~/.aws/**",
        "~/.gnupg/**",
        "**/.env",
        "**/.env.*",
        "**/*.pem",
        "**/*.key"
      ]
    },
    "excludedCommands": []
  },
  "permissions": {
    "allow": [
      "Bash(git *)",
      "Bash(npm *)",
      "Bash(node *)",
      "Bash(ls *)",
      "Bash(cat *)"
    ],
    "ask": [
      "Bash(*)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(rm -fr *)",
      "Bash(sudo *)",
      "Bash(dd *)",
      "Bash(mkfs *)",
      "Bash(git push --force*)",
      "Bash(git push *--force*)",
      "Bash(git reset --hard*)",
      "Read(~/.ssh/**)",
      "Read(~/.aws/**)",
      "Read(~/.gnupg/**)",
      "Read(**/.env)",
      "Read(**/.env.*)",
      "Read(**/*.pem)",
      "Read(**/*.key)",
      "Edit(~/.bashrc)",
      "Edit(~/.zshrc)",
      "Edit(~/.ssh/**)"
    ]
  }
}
EOF
ℹ️

各設定の意図：
enableAllProjectMcpServers: false — リポジトリの .mcp.json に仕込まれた悪意ある MCP サーバーの自動読み込みを防ぎます。MCP はサンドボックスの外で動くため、自動有効化すると SSH キーや .env を外部送信できてしまいます。
sandbox.filesystem.denyRead — OS レベルで機密パスへの読み取りを遮断します。サンドボックスが有効なとき、Bash コマンド（cat ~/.ssh/id_rsa 等）による迂回も防止します。
permissions.deny — 内蔵ツール（Read / Edit）への防御線です。sandbox.filesystem.denyRead と同じパスを両方に設定することで多層防御になります。ただし deny ルールが無視されるバグの報告が複数あるため（GitHub issues #6699, #24846, #27040）、sandbox.filesystem.denyRead を主防御、permissions.deny を副防御として扱ってください。

⚠️

permissions.deny の限界： sandbox なしでは Bash コマンドが deny ルールを完全に迂回できます（cat ~/.ssh/id_rsa は素通りします）。sandbox と deny を両方設定して初めて多層防御になります。sandbox が有効なとき、OS レベルで同じパスが遮断されるため、どちらか一方がバグで機能しなくても残りが守ります。

✅

settings.json の作成はここで完了です。STEP 6 で認証を済ませた後、セッション内から動作確認を行います。

STEP 6

認証（ログイン）

必須
⚠️

ネイティブインストール済みの場合：Windows側でログイン済みでも、WSL内では別途ログインが必要です。同じアカウントで問題ありません。

Claude Code 起動・認証WSL Ubuntu
claude

ブラウザが自動で開かない場合（WSLからWindowsブラウザへの連携ができていない場合）：

ブラウザが自動で開かない場合、ログイン画面で c キーを押すと URL がクリップボードにコピーされます。それを Windows のブラウザに貼り付けて認証してください。または以下で BROWSER 環境変数を設定しておくと自動で開くようになります。

BROWSER 環境変数を .bashrc に追記WSL Ubuntu
echo 'export BROWSER="/mnt/c/Program Files/Google/Chrome/Application/chrome.exe"' >> ~/.bashrc
.bashrc を反映WSL Ubuntu
source ~/.bashrc

ブラウザで認証完了後、表示されるトークンをターミナルに貼り付けて Enter。Claude Code のプロンプト（> の表示）が出れば認証成功です。

認証が完了したセッション内で、settings.json とサンドボックスが正しく有効になっているか確認します。

settings.json の構文チェック（シェルで実行）WSL Ubuntu
cat ~/.claude/settings.json | python3 -m json.tool

エラーなく整形表示されれば JSON として正しい形式です。続けて Claude Code セッション内でサンドボックス状態を確認します。

ℹ️

/sandbox と /doctor は Claude Code のスラッシュコマンドです。シェルプロンプトから実行すると「No such file or directory」になります。必ず claude で起動したセッション内（> の表示）から入力してください。

サンドボックスの有効確認（「Sandbox: enabled」と表示されれば完了）Claude Code セッション内
/sandbox
インストール状態・設定の総合チェックClaude Code セッション内
/doctor
✅

サンドボックス有効時、Bash コマンドは allowedDomains に列挙したドメイン以外への通信がブロックされます。Docker を使う場合は allowUnixSockets に "/var/run/docker.sock" を追加してください。

確認が終わったら /exit または Ctrl+C でシェルに戻り、STEP 7 へ進みます。

STEP 7

Git の設定（WSL内）

必須
ℹ️

Windows側に Git for Windows が入っていても、WSL内では Linux 版の Git を別途設定する必要があります。

まず Git が入っているか確認します。入っていなければインストールします。

Git バージョン確認WSL Ubuntu
git --version
Git インストール（入っていない場合のみ）WSL Ubuntu
sudo apt install -y git

GitHub アカウントと合わせてユーザー情報を設定します。

ユーザー情報の設定WSL Ubuntu
git config --global user.name "Your Name"
git config --global user.email "[email protected]"

デフォルトブランチを main に設定し、Windows 混在環境での改行コード変換を無効化します。

デフォルトブランチと改行コードの設定WSL Ubuntu
git config --global init.defaultBranch main
git config --global core.autocrlf false
STEP 8

GitHub HTTPS 連携

必須

HTTPS + PAT（Personal Access Token）を使って GitHub に接続します。credential helper でトークンを保存しておけば、2回目以降の push/pull でパスワード入力は不要です。

まず GitHub で PAT を発行します。

  1. 1. GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new token を開く
  2. 2. Note に適当な名前（例：WSL2-Ubuntu）を入力し、有効期限を設定
  3. 3. スコープの repo にチェックを入れて生成
  4. 4. 表示されたトークン（ghp_...）をコピー — 画面を閉じると再表示されません

次に、credential helper を設定してトークンをディスクに保存します。

credential helper の設定WSL Ubuntu
git config --global credential.helper store

プロジェクトディレクトリを作成してリポジトリをクローンします。初回のクローン時に Username と Password を求められます。

projects ディレクトリを作成してクローンWSL Ubuntu
mkdir -p ~/projects
cd ~/projects
git clone https://github.com/yourname/yourrepo.git
ℹ️

認証を求められたら Username に GitHub ユーザー名、Password に発行した PAT（パスワードではなくトークン）を入力してください。
入力内容は ~/.git-credentials に保存され、以降の push/pull では入力不要になります。

🚫

プロジェクトは必ず ~/projects/ 以下に置いてください。
/mnt/c/ 配下（Windows ドライブ上）に置くとファイルI/Oが約9倍遅くなり、Claude Code のパフォーマンスが大幅に低下します。

Git for Windows 側の PAT 保存（任意）

Windows 側（PowerShell や Git Bash）でも同じリポジトリを操作する場合は、Git for Windows の credential helper に PAT を保存します。Git for Windows をインストールすると Git Credential Manager（GCM） が標準で組み込まれており、Windows 資格情報マネージャーに安全に保存できます。

credential helper の確認Windows（PowerShell）
git config --global credential.helper

manager と表示されれば GCM が有効です。未設定の場合は以下で有効化します。

GCM を有効化（未設定の場合のみ）Windows（PowerShell）
git config --global credential.helper manager

設定後、初回の git clone または git push 時にブラウザが開いてGitHub認証画面が表示されます。ログイン後、PAT は Windows 資格情報マネージャーに自動保存され、以降は入力不要です。

ℹ️

ブラウザ認証ではなく PAT を直接入力したい場合は、Username に GitHub ユーザー名、Password に PAT を入力してください。GCM が Windows 資格情報マネージャーに保存します。
保存済みの認証情報は コントロールパネル → 資格情報マネージャー → Windows 資格情報 から確認・削除できます。

STEP 9

Windows Terminal の設定（任意・推奨）

推奨

Windows Terminal（タスクバーの「ターミナル」アプリ）でUbuntuをデフォルトに設定します。

  1. 1. Windows Terminal を開き Ctrl + , で設定を開く
  2. 2. 左メニュー「スタートアップ」→「既定のプロファイル」を Ubuntu に変更
  3. 3. 保存して閉じる

次回からターミナルを開くと直接 Ubuntu（WSL2）に入ります。

ℹ️

デフォルトのまま（PowerShell）でも、タブ横の ∨ ボタンから Ubuntu を選べます。どちらでも問題ありません。

STEP 10

Zed / VS Code のターミナルを WSL2 に向ける（任意）

推奨

Zed や VS Code の組み込みターミナルはデフォルトで Windows 側のシェルを開くため、claude コマンドが見つかりません。以下の設定を追加すると、エディタのターミナルが直接 WSL2 Ubuntu を起動し、WSL2 内の claude をそのまま使えるようになります。

Zed

~/.config/zed/settings.json（または Ctrl+, で開く設定ファイル）に以下を追加します。

Zed — settings.json に追加Windows
{
  "terminal": {
    "shell": {
      "with_arguments": {
        "program": "wsl.exe",
        "args": ["-d", "Ubuntu"]
      }
    }
  }
}
VS Code

VS Code のユーザー設定（settings.json）に以下を追加します。Ctrl+Shift+P →「Preferences: Open User Settings (JSON)」で開けます。

VS Code — settings.json に追加Windows
{
  "terminal.integrated.profiles.windows": {
    "WSL Ubuntu": {
      "path": "wsl.exe",
      "args": ["-d", "Ubuntu"]
    }
  },
  "terminal.integrated.defaultProfile.windows": "WSL Ubuntu"
}
ℹ️

設定後、エディタのターミナルを新規に開くと Ubuntu プロンプトが表示されます。claude --version で動作を確認してください。既存のターミナルタブは再起動が必要です。
