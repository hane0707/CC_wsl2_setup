概要

評価の目的と軸

「セキュリティ観点から堅実に確実な設計・実装を積み上げる」という基準で、Claude Code で利用可能な3つのプラグインを評価する。エージェント自律実行の利便性と、予測可能性・安全性のトレードオフを明示し、プロジェクト特性に応じた採用構成を導く。

評価5軸
#評価軸定義重要度
1 セキュリティガード力 シークレット検出・権限管理・ランタイムガードの充実度 高
2 プロセス規律 TDD強制・設計先行・検証強制の有無 高
3 予測可能性 エージェント行動の制御性・自律暴走リスクの低さ 高
4 サプライチェーンリスク 外部依存度・更新頻度・コード監査可能性（低リスク = 高スコア） 中
5 運用コスト 学習コスト・設定維持・バージョン追従の負荷（低コスト = 高スコア） 中
ℹ️

スコアは ●●●●●（5/5 最良）〜 ●○○○○（1/5 最低）の5段階。サプライチェーンリスク・運用コストは 低いほど高スコア。

P1

superpowers

公式プラグイン

哲学: 開発プロセスの体系化。 TDD・YAGNI・DRY原則をエージェントに強制するスキルセット。実装より設計を先行させ、「未検証の完了宣言」「根拠なき修正」を厳しく禁じる。マルチプラットフォーム対応（Claude Code・Cursor・Codex・Gemini CLI）。全13スキルで構成されシンプル。

主要スキル（全スキルが安全）
brainstorming writing-plans executing-plans test-driven-development systematic-debugging verification-before-completion subagent-driven-development requesting-code-review receiving-code-review dispatching-parallel-agents finishing-a-development-branch using-git-worktrees writing-skills
採用・不採用の論点

採用論点

  • TDD（RED→GREEN→REFACTOR）を強制し、テストのないコードを禁止
  • verification-before-completion で未検証の完了宣言を禁止
  • systematic-debugging で根拠なき修正を禁止
  • サブエージェントにセッション履歴を渡さないコンテキスト分離設計
  • YAGNI強制で不必要な複雑性・攻撃面を削減
  • 公式プラグイン（claude-plugins-official）でSCリスク最小
  • 13スキルで学習・管理コストが低い

懸念点

  • シークレット検出・権限管理の専用機能を持たない
  • 自律実行モードがなく生産性面では他に劣る
  • ランタイムセキュリティフックは別途設定が必要
強制ワークフロー（HARD-GATE）

superpowers の最大の特徴は、スキル間に HARD-GATE（通過しないと次に進めない検問）を設けている点。

フェーズスキルHARD-GATE の内容
設計 brainstorming 設計を提示しユーザーが承認するまでコードを1行も書かない
計画 writing-plans 実装計画をファイルに書き出し、ユーザーがレビューするまで実装しない
実装 test-driven-development テストが実際に失敗することを確認してから実装コードを書く（RED確認必須）
完了 verification-before-completion 検証コマンドの実行結果を示さずに「完了した」と宣言できない
デバッグ systematic-debugging 根本原因を特定する前に修正コードを書けない（推測での修正禁止）
✅

堅実実装との親和性: 最高。 セキュリティガードは薄いが、「間違いを未然に防ぐプロセス」の強制力は4プラグイン中で突出。HARD-GATEにより、エージェントが勝手に実装を進める・未検証で完了を宣言するという典型的なリスクを構造的に排除する。全構成で基盤として採用すべき唯一のプラグイン。

P2

oh-my-claudecode

コミュニティ

哲学: エージェント火力の最大化。 autopilot・ralph・ultrawork など複数の自律実行モードで高スループットを実現。Claude・Codex・Gemini の3モデル連携も可能。スキル数37と豊富で頻繁に更新される。superpowers との最大の違いは「プロセスを守る」より「タスクを完遂する」ことを優先する点。

スキル分類（セキュリティ観点）

緑: 安全に使用可能  /  黄: 使用前に目的を確認  /  赤: 堅実実装では原則不使用

ralplan verify trace deep-interview plan wiki remember ralph ultraqa team deepinit autopilot ultrawork ccg self-improve
採用・不採用の論点

採用論点（分析系スキル）

  • ralplan: 曖昧タスクを自律実行前にプランナー・アーキテクト・クリティックの3者合意で精査
  • verify: 完了宣言前の確認強制（superpowersと補完）
  • trace: エビデンス駆動の競合仮説並列追跡でデバッグ精度向上
  • deep-interview: 数学的曖昧さゲーティングで要件を結晶化
  • pre-tool-enforcer・permission-handler フックがツール実行前に動作

懸念点

  • autopilot/ralph/ultrawork: 権限・影響範囲が広く行動が予測困難
  • 外部GitリポジトリキャッシュによるSCA（サプライチェーン攻撃）リスク
  • 高頻度アップデートで設定が突然無効化されるリスク
  • sandbox 外で動作する旨がドキュメントに明記されている
  • 37スキルで学習・管理コストが高い
懸念点の補足説明
懸念事項具体的な意味対策
SCAリスク Claude Code はプラグインを ~/.claude/plugins/cache/ に Git クローンして使用する。oh-my-claudecode は非公式リポジトリのため、リポジトリへの不正コミット・タグ改ざん等でマルウェア入りスキルが自動配布されるリスクがある 定期的に git log でコミット履歴を確認する。メジャー更新前はスキルの差分をレビューする
sandbox外動作 settings.json の sandbox はBashツールの実行のみに適用される。oh-my-claudecodeのスキルが呼び出す Read・Edit・WebFetch・MCP ツールにはsandbox制限がかからない。ドキュメントにも「プラグインのコードがsandbox外で動作する」と明記 permissions.deny で機密ファイルへのRead/Edit/Writeを禁止し、sandbox非適用ツールの補完とする
pre-tool-enforcer の実態 ツール実行前に権限チェックを行うフック。ただし oh-my-claudecode の設定に基づいた独自ルールであり、settings.json の permissions.deny とは別系統。両方設定しておくことで多層防御になる oh-my-claudecodeのフックに頼らず、settings.jsonのpermissions.denyを必ず設定する
⚠️

堅実実装との親和性: 条件付き。 ralplan / verify / trace / deep-interview / plan は安全かつ有用。自律実行モード（autopilot / ralph / ultrawork）は堅実実装では原則不使用。使用する場合は明示的な承認と影響範囲の事前確認を必須とする。

P3

everything-claude-code

実験的

哲学: 包括的パフォーマンス最適化。 Anthropicハッカソン優勝を起点とした大規模システム。エージェント47体・スキル181個・ルール34個・フック8種以上。特にAgentShieldと呼ぶ統合セキュリティ機能がユニーク。他の2プラグインにはないシークレット検出・ランタイムブロック機能を持つ。

AgentShield — 2層の保護構造

AgentShieldは静的解析（事前検査）とランタイムフック（実行時ブロック）の2層で構成される。

層タイミング機能評価
層1 静的解析 セッション開始前 102ルール・1282テストでCLAUDE.md・settings.json・MCPコンフィグ・hooksを走査。権限過剰付与・フックインジェクション・MCPリスクを検出 有用
層2 ランタイムフック① プロンプト送信直前 beforeSubmitPrompt フック: sk-・ghp_・AKIA等14パターンのシークレットをスキャン。含まれていれば送信をブロック ★ ユニーク
層2 ランタイムフック② ファイル読み取り直前 beforeTabFileRead フック: .env・.key・.pemへのアクセスをブロック。permissions.denyが効かないエディタ系ツールの補完として機能 ★ 有用
既存のsandbox / permissions.deny との違い
手段適用タイミングカバー範囲カバーできないもの
sandbox.filesystem Bashコマンド実行時 Bashツール経由のファイルアクセス Read/Edit/Glob/Write ツール
permissions.deny 各ツール呼び出し時 Read・Edit・Glob・Write・Bash プロンプト内容のシークレット
AgentShield フック プロンプト送信前・ファイル読取前 プロンプトに含まれるシークレット・ファイル読取 Bash経由のネットワーク通信
ℹ️

AgentShieldが防ぐ固有のリスク: permissions.deny はファイルへの「読み取りアクセス」を防ぐが、ユーザーが手動でAPIキーをプロンプトに貼り付けた場合や、環境変数がプロンプトに展開されてしまった場合は防げない。AgentShieldの beforeSubmitPrompt フックはプロンプト送信直前のシークレット検出により、この「最後の砦」となる。

採用上の注意点（フック名について）
⚠️

beforeSubmitPrompt・beforeTabFileRead は everything-claude-code が独自に定義したフック名称。標準Claude Code のフックイベント名（PreToolUse・PostToolUse・Stop等）とは異なる。
採用前に必ず以下を確認すること:
① リポジトリから実際のスクリプトを取得し、フックの実装コードを自分でレビューする
② Claude Code のバージョンでこれらのフックイベントが有効かを確認する
③ plugin.json に hooks フィールドを追加しない（v2.1+ では hooks/hooks.json の自動読み込みと重複してエラーになる）

採用・不採用の論点

採用論点

  • AgentShieldのシークレット検出は3プラグイン中で唯一の機能
  • .env/.key/.pemのランタイムブロックはpermissions.denyの多層防御として有効
  • フックインジェクション分析・MCPリスク評価はユニーク
  • 1282テスト・102静的解析ルールの量的充実

懸念点

  • ハッカソン起点でエンタープライズ実績が未検証
  • ルールは手動インストールが必須（自動配布不可）
  • plugin.jsonのhooksフィールド制約など運用上の落とし穴（Issue #29/#52/#103）
  • エージェント47体・スキル181個で管理コストが甚大
  • Claude Code CLI v2.1.0以降が必須
ℹ️

堅実実装との親和性: AgentShieldのみ高。 プラグイン全体の採用はリスクが高い。AgentShieldのシークレット検出とランタイムガード部分のみを手動設定する部分採用が現実的な選択。

P4

snarktank/ralph

コミュニティ（★15.5k）

哲学: 完了するまでループで回し続ける。 Geoffrey Huntley が考案した「Bashループ自律コーディング」パターンの実装。本質は while :; do cat PROMPT.md | claude; done の1行に集約される。各反復でAIインスタンスを新規起動してコンテキスト汚染を回避し、記憶は git history・progress.txt・prd.json の3ファイルだけで管理する。

oh-my-claudecodeの ralph スキルとは独立した別実装。Star数15,500・Fork数1,600と他ツールに比べ異例の規模を持ち、Claude Plugin Marketplace でも公式提供（インストール数129,572）されている。

ファイル役割

セキュリティ上の重大な注意点
🚨

Claude Code モードでは --dangerously-skip-permissions が必須。 全ツール呼び出し確認ダイアログが無効化され、AIが任意のファイル操作・コマンド実行を無制限に行える状態になる。

実際に起きた事故（記録済み）:
· 2025年12月: rm -rf tests/ patches/ ~/ の末尾 ~/ が展開され、デスクトップ・Keychain・アプリデータが消滅
· 2025年11月: ~ という名前のディレクトリを誤作成後、後続の rm -rf * がホームディレクトリを削除

採用・不採用の論点

採用論点

  • 要件と完了条件が明確なタスク（テストが通る等）に対して強力
  • 各反復でフレッシュなコンテキストを使うためコンテキスト汚染を回避
  • bash + jq だけで動く最小構成。学習・導入コストが最低
  • Amp CLI / Claude Code CLI どちらでも動くツール非依存設計
  • グリーンフィールド開発の夜間自律実行（Y-CombinatorHackで一晩6リポジトリの実績）

懸念点

  • --dangerously-skip-permissions 必須 → サンドボックス外での実行は事故の元
  • ガード機能ゼロ。ralph.sh 自体に危険なコマンドを検出する仕組みなし
  • セキュリティセンシティブなコード（認証・暗号・ACL）を書かせるのは高リスク
  • 完了シグナル（<promise>COMPLETE</promise>）は単純なgrep一致で信頼性が低い
  • メンテナー体制が不安定（Issue #107/#108 でメンテナー不在問題が議論中）
  • 50反復ループで大規模コードベースは $50〜$100 超えも現実的

⚠️

ralph の位置づけ
🔄

ralph は上記3構成とは異なる次元のツール。 プラグインというより「実行エンジンパターン」に近い。上記3構成はいずれも インタラクティブなClaude Codeセッション中に何をするか を定義するが、ralphは セッションそのものを自律的に反復する。

追加構成: バッチ自律実行型（サンドボックス限定）
採用: superpowers（PRD作成・品質基準）+ ralph（Dockerコンテナ内で夜間実行）
前提: Docker/Devcontainerによる完全隔離・本番資産の排除・max_iterationsの上限設定
用途: 大規模リファクタリング・テスト追加の一括適用・グリーンフィールド開発の夜間バッチ

ℹ️

2つのフローの主な差分: バランス型はSTEP 01（要件整理）とSTEP 05（/verify による二重確認）が追加される。デバッグ時に /trace による競合仮説並列追跡が使える点も差分。いずれのフローでも HARD-GATE と AgentShieldフックは常に有効。

推奨

推奨構成

★

結論: バランス型をベースに、AgentShieldで最終防衛線を張る3層構成。
superpowers でプロセス規律を確立し、oh-my-claudecode の分析力で設計精度を高め、AgentShield のフック2本でシークレット漏洩を防ぐ。各層が独立しているため、リスク許容度に応じて段階的に適用できる。

推奨採用スタック（バランス型）

SP

superpowers — 全スキル（第1層: プロセス規律の基盤）

TDD・YAGNI・verification-before-completion・systematic-debugging を常時適用。設計先行・未検証完了禁止・サブエージェントコンテキスト分離で「間違いを作らない構造」を維持する。公式プラグインのため追従コスト低。

OMC

oh-my-claudecode — 分析・計画系5スキルのみ（第2層: 設計精度の向上）

意識的に使用するスキル: ralplan・verify・trace・deep-interview・plan。
autopilot・ralph・ultrawork は明示的な承認なしに使用しない。高頻度アップデートへの追従コストを許容できない場合は本層を外し堅実重視構成へ切り替える。

ECC

everything-claude-code — AgentShieldフック2本のみ手動設定（第3層: シークレット防衛）

プロンプト送信前のシークレット検出フック（beforeSubmitPrompt）と、.env/.key/.pem のランタイムアクセスブロックフックの2本のみ手動設定。プラグイン全体は採用しない。設定前にスクリプトの内容を必ず自身でレビューすること。

採用判断サマリー
プラグイン採用範囲理由
superpowers 全スキル採用 公式プラグイン・プロセス規律最強・SCリスク最小・学習コスト低。全構成で基盤として採用。
oh-my-claudecode 分析系5スキルのみ 自律実行モードは予測困難・SCリスクあり。分析系は安全で価値が高い。バージョン追従が不要になったら再評価。
everything-claude-code AgentShield 2フックのみ 成熟度・管理コストの問題でフル採用リスク大。シークレット検出は4プラグイン中唯一の価値があり、この部分のみ手動設定で活用する。
snarktank/ralph 通常セッションでは不採用 --dangerously-skip-permissionsは堅実実装の前提（権限最小化）と根本矛盾。Docker等の隔離環境でのバッチ実行に限り、superpowersとの組み合わせで検討可。
